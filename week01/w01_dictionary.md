# 1주차 핵심 용어집

강의 자료의 비유와 설명을 바탕으로 정리한 용어집입니다. 처음 접하는 개념은 비유를 통해 먼저 감을 잡고, 정확한 정의로 마무리하세요.

---

## 1. 소프트웨어 개발 기초

**프레임워크 (Framework)**
> "골격이 이미 있다. 나는 살만 붙이면 된다."

애플리케이션 개발에 필요한 공통 기능과 흐름을 미리 구조화해 제공하는 소프트웨어입니다. 개발자는 정해진 규칙에 맞춰 코드를 작성하고, **제어권은 프레임워크**가 가집니다.
- 비유: 골조와 배관이 완성된 **모델하우스** (인테리어만 하면 됨), 매뉴얼대로 운영하는 **프랜차이즈**

---

**라이브러리 (Library)**
> "필요한 부품을 내가 골라서 쓴다."

개발자가 필요할 때 직접 호출해서 사용하는 기능(코드)들의 모음입니다. 흐름의 **제어권은 개발자**에게 있습니다.
- 비유: 직접 조립해야 하는 **IKEA 가구**, 꺼내 쓰는 **공구 세트**

> **프레임워크 vs 라이브러리 핵심 차이:** 제어권이 누구에게 있는가.
> 라이브러리는 내가 부르고, 프레임워크는 나를 부른다.

---

## 2. Spring 핵심 개념

Spring의 핵심은 **IoC/DI, AOP, PSA** 세 가지입니다. POJO는 이 모든 것을 가능하게 하는 전제 조건입니다.

---

**POJO (Plain Old Java Object)**
특정 프레임워크나 기술에 얽매이지 않은 **순수한 자바 객체**입니다. Spring은 POJO 기반으로 설계되어 코드가 간결하고, 다른 환경에서도 재사용하기 쉽습니다.

---

**IoC (제어의 역전, Inversion of Control)**
> "내가 프레임워크를 부르는 게 아니라, 프레임워크가 내 코드를 부른다."

프로그램의 **제어 흐름을 개발자가 아닌 컨테이너(프레임워크)가 담당**하는 원리입니다.
- 할리우드 원칙: **"Don't call us, we'll call you"** — 우리가 부를 테니 기다려라

---

**DI (의존성 주입, Dependency Injection)**
IoC를 코드로 실현하는 **구체적인 기법**입니다. 개발자가 코드 내부에서 직접 객체를 생성하지 않고, **Spring 컨테이너가 객체를 만들어 외부에서 주입**해 줍니다.

> IoC(원리) ⊃ DI(구현 방법) — IoC라는 개념을 실제로 구현하는 방식 중 하나가 DI입니다.

---

**AOP (관점 지향 프로그래밍, Aspect-Oriented Programming)**
로그, 보안, 트랜잭션처럼 여러 기능에 **공통으로 반복되는 코드(횡단 관심사)를 핵심 비즈니스 로직에서 분리**해 따로 관리하는 기법입니다. 핵심 로직이 깔끔해지고 유지보수가 쉬워집니다.

---

**PSA (이식 가능한 서비스 추상화, Portable Service Abstraction)**
내부 기술(데이터베이스 종류 등)이 바뀌더라도 **개발자가 작성한 코드는 수정하지 않아도 되도록** 중간에서 통일된 인터페이스를 제공하는 추상화 계층입니다.

---

**Spring Boot (스프링 부트)**
기존 Spring Framework의 복잡한 설정을 자동화하고, 웹 서버(Tomcat)를 내장하여 **설정 없이 바로 실행 가능한 실무 표준 프레임워크**입니다. 현대 Java 백엔드 개발의 시작점입니다.

---

## 3. 프로젝트 및 개발 방법론

**SI 프로젝트 (System Integration)**
고객의 요구사항 분석부터 기획 → 설계 → 개발 → 테스트 → 배포까지 **전 과정을 거쳐 정보 시스템을 통합 구축**해주는 프로젝트입니다.

---

**RFP (제안요청서, Request for Proposal)**
고객(발주처)이 "이런 시스템을 만들어 주세요"라고 요구사항을 상세히 기술한 문서로, **SI 프로젝트의 출발점**입니다.

---

**폭포수 방법론 (Waterfall)**

```
요구사항 분석 → 설계 → 구현 → 테스트 → 배포
```

단계를 순서대로 완료하고 다음 단계로 넘어가는 방식입니다. 이전 단계로 돌아가기 어렵지만 일정 예측이 명확해 **공공·대규모 SI 프로젝트**에 많이 사용됩니다.

---

**애자일 방법론 (Agile)**

```
[1~4주 Sprint] 계획 → 개발 → 피드백 → 반복
```

짧은 반복 주기(Sprint)로 **계속 개발하고 피드백을 반영**하는 유연한 방식입니다. 요구사항이 자주 바뀌는 스타트업이나 웹/앱 서비스 개발에 적합합니다.

> **Waterfall vs Agile 핵심 차이:** 변화에 얼마나 유연하게 대응할 수 있는가.
> Waterfall은 계획 중심, Agile은 변화 중심.
